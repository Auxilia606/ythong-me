# `shared` Layer Guide

## Role

`shared` is the generic foundation layer.

It contains reusable code with no business meaning.

## Put Code Here When

- the code is domain-agnostic
- multiple unrelated domains can use it
- the main value is reuse of infrastructure or primitives
- the module does not encode product-specific business rules

## Typical Contents

- design-system components
- API client infrastructure
- formatting and date utilities
- environment and runtime config
- theme tokens
- low-level library wrappers

## Typical Structure

```text
src/shared/
  api/
  config/
  lib/
  theme/
  ui/
```

`shared` does not use business slices.

## API Client Modules

`src/shared/api` may contain multiple client modules when they represent
different infrastructure responsibilities rather than different business layers.

- `fetch-with-auth.ts` is the low-level authenticated `fetch` primitive.
  It is responsible for applying the shared authorization header behavior and
  can be exported from `src/shared/api/index.ts` for manual transport use.
- `orval-fetch-client.ts` is the Orval mutator used by generated API code.
  It adapts generated endpoint functions to this app's transport behavior:
  resolving API URLs, calling `authorizedFetch`, parsing response bodies, and
  throwing Orval-compatible typed errors.
- Generated API files under `src/shared/api/generated` should call the Orval
  mutator. Application code should normally consume generated endpoint hooks or
  slice-level API wrappers instead of importing `orvalFetchClient` directly.

Keep the direction of dependency one-way:

```text
generated API -> orval-fetch-client.ts -> fetch-with-auth.ts
```

Do not merge these modules unless the project no longer uses Orval-generated
clients or no longer needs a standalone authenticated fetch primitive.

## Orval-Generated API

`src/shared/api/generated` is the output directory for Swagger/OpenAPI-based
client code generated by Orval.

Generation is configured in `orval.config.ts` and run with:

```bash
pnpm api:generate
```

Current output shape:

```text
src/shared/api/generated/
  patent-radar.ts
  model/
```

Rules:

- treat everything under `src/shared/api/generated` as generated source
- do not hand-edit generated files for application behavior
- expect files, exported names, DTOs, query hooks, and mutation hooks to be
  added, removed, or renamed whenever the backend Swagger contract changes
- fix API shape issues in the backend Swagger contract or Orval configuration,
  then regenerate the client
- keep custom transport behavior in `orval-fetch-client.ts`, not inside the
  generated files
- keep manually authored wrappers, mappers, or compatibility code outside the
  generated directory

Application code may import generated hooks, endpoint functions, and model types
when it is acceptable to follow the backend contract directly. When a screen or
feature needs a more stable frontend-facing API, create a slice-level API module
in the relevant `features/*/api`, `entities/*/api`, `widgets/*/api`, or
`pages/*/api` boundary and map generated DTOs there.

Generated models are transport DTOs. Do not add product rules, derived fields,
or UI-specific formatting to generated model types. Put that logic in the
consuming slice's `model`, `api`, or `lib` segment depending on ownership.

Review generated diffs as backend-contract diffs:

- endpoint addition or removal means the Swagger contract changed
- request or response type changes may require frontend call-site updates
- changes in query keys, hook names, or parameter order can affect React Query
  cache behavior
- deleting or renaming generated exports should be handled by updating imports
  or adding explicit slice-level compatibility where needed

## Keep Out of `shared`

- product rules
- business terms that belong to a feature or entity
- route orchestration
- hidden feature logic extracted only for convenience

## Import Rules

- `shared` must not depend on upper layers
- keep APIs generic and explicit
- prefer small, focused modules over large helper buckets

## Guardrail

Before moving code into `shared`, ask:

1. Does this code still make sense without product-specific context?
2. Could two unrelated domains use it without renaming or rewriting?
3. Would moving it here hide important business meaning?

If any answer is no, keep the code in its current slice.

## Anti-Patterns

- `shared/utils` becoming a dumping ground
- generic-looking helpers that actually encode business rules
- moving code into `shared` only to bypass layer boundaries
